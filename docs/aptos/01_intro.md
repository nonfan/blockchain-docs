# 公链概述
> Aptos 的架构、执行模型、账户与资源、工具链与生态一览

## 核心目标
- 安全的资源模型（Move）
- 可扩展的并行执行（低延迟、高吞吐）
- 友好的开发工具与生态集成

## 架构与执行
- 并行执行：通过 Block-STM 将可并行事务在同一区块并发执行，减少冲突、提升吞吐
- 事务类型：`EntryFunction`（入口函数调用）与 `View`（只读函数），以 BCS 编码在链上执行
- 状态存储：以账户地址为根的资源树，模块与资源挂载在账户下，状态以可验证结构持久化

## 共识与网络
- 共识机制：现代 BFT 的管线化设计，追求快速最终性与高吞吐
- 网络环境：`mainnet` / `testnet` / `devnet`，测试网提供 Faucet 与开放节点

## Move 与资源模型
- 线性资源：资产不可随意复制或丢弃，需显式移动与销毁
- 能力系统：`copy / drop / store / key` 控制类型行为与存储形态
- 权限控制：通过能力对象与模块接口限制敏感操作（如铸造、销毁、升级）

## 账户与交易
- 账户模型：每个地址持有资源与模块，入口函数对账户资源进行读写
- 交易流程：构造负载 → 模拟估算 → 签名提交 → 等待确认（推荐）
- 费用与性能：Gas 计费与并行执行结合，降低平均确认延迟

## 开发与工具链
- CLI / SDK：官方 CLI、TypeScript SDK 与 Rust SDK，支持查询、交易与账户管理
- 前端钱包：`@aptos-labs/wallet-adapter-react` 集成主流钱包（如 Petra、Pontem）
- 数据索引：Indexer GraphQL 提供高层查询接口，便于检索账户、交易、事件等信息

## 生态与应用
- 典型方向：支付与转账、NFT 收藏、DeFi 协议、游戏与链上身份
- 开发路径：从最小合约与转账开始，逐步接入钱包、索引器与前端交互，迭代为生产级应用

## 与 Sui 的差异（开发视角）
- 数据模型：Aptos 以账户挂载资源，Sui 以对象与所有权为核心；同为 Move，但建模方式不同
- 并行执行：两者均重视并行与低延迟；调用与权限设计在各自生态中有不同标准组件与模式
